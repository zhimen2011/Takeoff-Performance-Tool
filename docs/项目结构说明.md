# 项目结构说明

本文档说明 STAS 起飞性能计算工具重构版的推荐项目结构、模块职责和后续扩展方式。

## 1. 当前状态

当前项目仍处于重构准备阶段。

旧版软件资料位于：

```text
examples/STAS old/
```

旧版主程序为：

```text
examples/STAS old/TOPC_PCUI_V2.py
```

旧版代码暂时保留，不直接删除。新版业务代码应逐步迁移到 `src/`，并通过测试验证。

## 2. 新版支持范围

新版第一阶段只支持：

- `738`
- `777F`

旧版 `733` 机型从新版功能中彻底移除，但旧资料仍保留在 `examples/`，用于历史追溯或人工参考。

## 4. 根目录文件

- `README.md`：项目总入口，说明安装、运行、结构和常见问题。
- `AGENTS.md`：AI 协作规则。
- `run_stas.py`：仓库根目录联调启动脚本，自动把 `src/` 加入导入路径。
- `run_desktop.py`：仓库根目录桌面界面启动脚本。
- `STAS_CODE_HANDOFF_2026-05-16.md`：当前任务队列改造交接记录，便于下次直接接续。
- `docs/`：长期维护文档。
- `examples/`：旧版资料和示例数据。
- `src/`：新版业务代码。
- `tests/`：自动化测试。
- `config/app.example.toml`：应用级配置模板，用于本地真实 STAS 联调。

## 5. docs 目录

`docs/` 存放项目长期文档：

- `需求说明.md`：记录目标、范围、功能、验收标准。
- `项目结构说明.md`：记录目录职责和扩展规则。
- `数据流说明.md`：记录输入、处理、输出、异常流。
- `开发记录.md`：记录阶段性变更和技术决策。

每次新增核心功能后，应同步更新相关文档。

## 6. src/stas_app/ui

界面层。

职责：

- 显示机型、机场、跑道、温度、风、QNH 和 QNHREF 输出标注等输入控件。
- 显示道面条件、污染深度、引气/空调、防冰和推力（顺序项）控件。
- 管理跑道复选框选择，包括全选、全不选和反选。
- 管理输出顺序队列，包括加入当前顺序项、加入默认四项、上移、下移、删除、清空和执行。
- 管理队列方案存档，包括保存、加载和删除；队列方案不保存机型、机场、跑道和报告格式。
- 管理手册起飞分析格式选择；该选择属于报告输出，不属于输出顺序项，并支持“不生成手册格式”。选择具体手册格式时，临时起飞分析格式不再同时生成。
- 展示计算状态、耗时、结果路径和错误提示。
- 调用服务层完成业务动作。

禁止：

- 直接解析 `APTRWY.RWY`。
- 直接拼接 STAS 输入文件。
- 直接调用 `STAS.exe`。
- 直接生成 Word/PDF 报告。
- 在 UI 中硬编码机型业务规则。

当前已实现：

- `src/stas_app/ui/desktop_app.py`
- `src/stas_app/ui/forms.py`
- `run_desktop.py`

当前桌面界面已改用 DearPyGui，不再依赖 `tkinter`。当前已加入 Scenario 队列和队列方案存档功能。本轮已完成不打开窗口的控件构建检查，后续仍需人工点击验证完整计算流程。

当前布局将飞行参数、基础计算条件和当前输出顺序项输入控件放在左侧，将报告输出、已保存队列方案、当前输出顺序和结果区依次放在右侧。温度、风、QNH 和 QNHREF 输出标注是可保存的计算条件；道面条件、推力、引气和防冰是顺序项字段；手册起飞分析格式是报告输出字段；污染深度随道面条件启用或禁用。报告输出区域同时承载“加入当前顺序项”“加入默认四项”“只算当前”和“重置”操作按钮。当前输出顺序列表会压缩多跑道显示，例如 `RWY 10L/10R/+3`。

## 7. src/stas_app/models

数据模型层。

建议模型：

- `AircraftConfig`：机型配置。
- `AppConfig`：应用级路径和运行配置。
- `PerformanceRequest`：用户计算请求。
- `PerformanceResult`：计算结果。
- `RunwayInfo`：跑道信息。
- `AirportInfo`：机场信息。

作用：

- 明确输入输出字段。
- 降低字典传参导致的错误。
- 为后续测试和扩展提供稳定结构。

## 8. src/stas_app/parsers

解析层。

第一阶段主要包含：

- `runway_parser.py`：解析 `APTRWY.RWY`。
- `stas_output_parser.py`：解析 STAS 输出中的性能结果行，为后续队列汇总和报表合并提供结构化数据。

后续可扩展：

- STAS 输出解析。
- OFP 解析。
- NOTAM 解析。
- Excel/CSV 参数解析。

解析层只负责把原始数据变成结构化数据，不负责业务判断。

## 9. src/stas_app/services

核心业务层。

建议模块：

- `app_factory.py`：根据应用配置组装完整计算服务。
- `aircraft_registry.py`：读取并管理机型配置。
- `input_builder.py`：根据请求参数和模板生成 STAS 输入文件。
- `performance_service.py`：对外提供单次计算入口，串联校验、输入生成、STAS 执行和报告导出。
- `runway_import_service.py`：预览外部 `.RWY` / `.rwy` / `.stx` 文件并导入机场完整块到 `APTRWY_MASTER.RWY`。
- `runway_intersection_generator.py`：对 AIRPORT2 `#INT` 交叉口生成短跑道块，合并多行 `H ` EOSID 文本，并清理尾部航空日期。
- `runway_procedure_enricher.py`：报告导出前读取运行用 `APTRWY.RWY`，为 STAS 截断的 AIRPORT2 特殊程序生成完整报告输出；包含 `NO EMERGENCY TURN` 的跑道不补全。
- `runway_runtime_file.py`：计算前从 `APTRWY_MASTER.RWY` 复制选中机场完整块，生成 STAS 实际读取的 `APTRWY.RWY`；写入前会套用交叉口短跑道生成逻辑。
- `scenario_options.py`：管理机型相关 Scenario 选项映射，例如 738/777F 的引气、防冰、道面条件和污染深度规则。
- `stas_automator.py`：提供任务队列自动化基础能力，用于按 Scenario 列表生成输入、执行和解析结果。
- `stas_engine.py`：调用 `STAS.exe` 并返回结构化结果。
- `validation.py`：校验机场、跑道、温度、风、QNH、推力设置。
- `smb_sync.py`：同步远端跑道数据文件。

业务层是后续新增功能的主要扩展点。
后续 UI 层应优先调用 `performance_service.py`，避免界面直接依赖底层多个模块。

机场主库与运行跑道文件说明见：

```text
docs/机场主库与运行跑道文件.md
```

## 10. src/stas_app/exporters

导出层。

第一阶段包含：

- `queue_report.py`
- `manual_takeoff_report.py`
- `word_report.py`
- `pdf_report.py`

规则：

- 导出层只处理报告生成。
- Word/PDF 失败不应覆盖 STAS 核心计算成功状态。
- 后续增加 Excel、CSV、JSON 报告时，应放在该目录。

当前已实现：

- `src/stas_app/exporters/queue_report.py`
- `src/stas_app/exporters/manual_takeoff_report.py`
- `src/stas_app/exporters/word_report.py`
- `src/stas_app/exporters/pdf_report.py`
- `src/stas_app/models/report.py`

队列级合并报告由 `QueueReportExporter` 负责。它接收按用户顺序执行后的 `PerformanceCalculationResult` 列表，只合并成功且存在报告读取源的 Scenario；如果单次计算生成了 `STASOUT.enriched.out`，队列报告优先读取该文件，否则回退到 `STASOUT.out`。未选择手册格式时输出 `STAS_QUEUE.out`、`队列_临时起飞分析.docx` 和可选的 `队列_临时起飞分析.pdf`；选择手册格式时输出 `STAS_QUEUE.out` 和队列手册 Word/PDF，不再同时生成队列临时报告。

原有 `word_report.py` 输出现在称为“临时起飞分析格式”。`manual_takeoff_report.py` 输出“手册起飞分析格式”，从 `templates/reports/manual_takeoff/templates.toml` 读取模板注册表。当前手册模板包括 `738_normal`、`777f_normal`、`777f_derate`、`777f_bump`，后续新增格式时优先新增模板配置和 `.docx` 模板文件。桌面界面选择“不生成手册格式”时，服务层收到空的 `manual_report_template_id`，只生成临时 Word/PDF；选择具体手册格式时，只生成手册 Word/PDF。

## 11. src/stas_app/storage

存储层。

职责：

- 读取和保存用户配置。
- 管理输出目录。
- 保存运行元数据。
- 后续管理历史记录。

建议每次计算创建独立输出目录，避免覆盖旧结果。

示例：

```text
output/
  2026-05-13_153000_ZUUU_777F/
    STASINP
    STASOUT.out
    临时起飞分析.docx（未选择手册格式时）
    临时起飞分析.pdf（未选择手册格式时）
    手册起飞分析.docx（选择手册格式时）
    手册起飞分析.pdf（选择手册格式时）
    run_metadata.json
    error.log
```

当前已实现：

- `src/stas_app/storage/config_repository.py`
- `src/stas_app/storage/output_manager.py`
- `src/stas_app/storage/runway_master_repository.py`
- `src/stas_app/storage/scenario_order_store.py`

`config_repository.py` 负责读取 `config/app.local.toml` 或其他 TOML 配置文件，把相对路径解析成可执行路径，供服务组装层使用。

`scenario_order_store.py` 负责保存可复用的队列方案。默认运行时文件位于：

```text
output/scenario_orders.json
```

队列方案保存 `temperature_range`、`wind_range`、`qnh_ref`、`describe_qnh_ref`、`runway_condition`、`contamination_depth`、`thrust_option`、`derate`、`bleed`、`anti_icing`，不保存机型、机场、跑道，也不保存手册起飞分析格式。加载时使用当前界面的机型、机场、跑道和报告输出；温度、风、QNH、QNHREF 输出标注以及顺序项使用队列方案中保存的值。`derate` 目前主要用于兼容旧存档；GUI 新增队列项使用 `thrust_option` 表达推力选择。

`runway_master_repository.py` 负责 `APTRWY_MASTER.RWY` 的读取、备份、合并和写回。GUI 机场数据导入会通过该仓储更新主库，不直接写入 STAS 运行文件 `APTRWY.RWY`。AIRPORT2 交叉口短跑道只在内存数据集和运行文件生成时生成，主库仍保留外部文件的原始机场块。

## 13. templates

STAS 输入模板目录。

第一阶段只保留：

```text

templates/777F.inp

```

模板规则：

- 不写死 `APTRWY.RWY` 绝对路径。
- 使用统一占位符。
- 模板只表达 STAS 输入格式，不写 Python 逻辑。

手册起飞分析 Word 模板只表达版式、页眉和页面设置；正文由 `ManualTakeoffReportExporter` 用 STAS 输出重新写入。正文会保留 `      ELEVATION` 前导空格，每个 `ELEVATION` 段使用段首分页独立输出，并根据估算行数选择正常、紧凑或更紧凑的字号和行距，降低跨页和额外空白页风险。

建议占位符：

- `{airport_file}`
- `{airport_runway}`
- `{runway_condition}`
- `{contamination_depth}`
- `{bleed_status}`
- `{anti_icing}`

- `{wind_range}`
- `{qnh_ref}`
- `{qnh_ref_description}`：控制 `DESCRIBE $...$` 中是否标注 QNHREF；关闭时生成 `DESCRIBE $$`，但 `QNHREF = {qnh_ref}` 参数行仍保留。

## 15. 扩展原则

后续增加功能时遵守：

- 新机型优先通过配置和模板扩展。
- 新输入来源放在 `parsers/`。
- 新业务规则放在 `services/`。
- 新导出格式放在 `exporters/`。
- 新持久化需求放在 `storage/`。
- UI 只调用服务，不承载核心逻辑。
- 每个新增核心功能都要补测试和文档。

## 16. 2026-05-19 跑道过滤与交叉口拆分

本次更新涉及 `models/`、`services/`、`ui/` 和 `tests/` 四层。

- `models/runway.py`：`Runway` 增加 `tora_m` 和 `is_intersection`，用于描述跑道 TORA 和是否为非全跑道。

- `services/app_factory.py`：从生成后的机场块构建 UI 用 `RunwayDataset`，保留跑道编号、记录类型、TORA 和非全跑道标记。
- `ui/desktop_app.py`：跑道选择区支持勾选启用最小 TORA 过滤；机场跑道数据导入入口移入“机场跑道数据管理”折叠区。
