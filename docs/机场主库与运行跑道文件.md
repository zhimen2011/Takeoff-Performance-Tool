# 机场主库与运行跑道文件

本文档说明 APTRWY 主库和 STAS 运行文件的关系，以及 GUI 机场数据导入流程。

## 1. 文件角色

当前设计区分两个文件：

```text
runtime/stas/APTRWY_MASTER.RWY
runtime/stas/APTRWY.RWY
```

`APTRWY_MASTER.RWY` 是完整机场主库，供界面机场/跑道选择和校验使用。

`APTRWY.RWY` 是 STAS 实际读取的运行文件。程序会在计算前从主库复制选中机场的完整原始块，自动生成该文件。

如果主库文件尚不存在，系统会先回退到旧逻辑：界面和 STAS 都使用 `APTRWY.RWY`。之后可以通过 GUI 导入功能创建 `APTRWY_MASTER.RWY`。

## 2. 当前计算流程

```text
启动程序
  -> 优先读取 airport_runway_master_file
  -> 如果未配置但存在 APTRWY_MASTER.RWY，则读取该文件
  -> UI 显示主库中的机场和跑道

用户选择机场和跑道
  -> 界面只保存选择，不立即修改 APTRWY.RWY

用户点击“只算当前”或“执行队列”
  -> PerformanceService 校验机场/跑道
  -> RunwayRuntimeFileService 从 APTRWY_MASTER.RWY 提取当前机场完整块
  -> 备份当前 APTRWY.RWY
  -> 写入新的 runtime/stas/APTRWY.RWY
  -> STAS 按原逻辑读取 APTRWY.RWY
```

注意：当前只按机场复制完整块，不按跑道裁剪。即使用户只选择一个跑道，写入 `APTRWY.RWY` 的也是该机场完整原始块。

## 3. 配置项

示例：

```toml
[paths]
airport_runway_file = "runtime/stas/APTRWY.RWY"
airport_runway_master_file = "runtime/stas/APTRWY_MASTER.RWY"
```

字段含义：

- `airport_runway_file`：STAS 实际读取的运行文件。
- `airport_runway_master_file`：完整机场主库。

`airport_runway_master_file` 是可选项。未配置时，系统会把 `airport_runway_file` 同目录下的 `APTRWY_MASTER.RWY` 作为默认主库路径。主库文件不存在时仍可启动，导入功能会把该路径作为写入目标。

## 5. GUI 机场数据导入

导入入口在桌面界面的“机场数据导入”区域。

当前支持：

- 选择或输入外部 `.RWY`、`.rwy` 或 `.stx` 文件路径。
- 预览外部文件中可识别的机场。
- 判断机场在 `APTRWY_MASTER.RWY` 中是否已存在。
- 选择“跳过已存在机场”或“覆盖已存在机场”。
- 导入前自动备份已有 `APTRWY_MASTER.RWY`。
- 导入完成后刷新 GUI 机场/跑道列表。

导入目标固定为：

```text
runtime/stas/APTRWY_MASTER.RWY
```

不应直接把外部机场数据导入到：

```text
runtime/stas/APTRWY.RWY
```

如果本地还没有 `APTRWY_MASTER.RWY`，导入时会用当前 `APTRWY.RWY` 作为初始主库，再把外部机场合并进去。这样可以避免第一次导入时丢掉现有机场。

数据流：

```text
外部 .RWY / .rwy / .stx
  -> runway_database_parser 保留机场完整原始块
  -> RunwayImportService.preview_import 生成预览
  -> RunwayMasterRepository.load_or_seed 读取主库或从 APTRWY.RWY 初始化
  -> 选择跳过/覆盖策略
  -> 备份 APTRWY_MASTER.RWY
  -> 写回 APTRWY_MASTER.RWY
  -> 重新加载 ApplicationContext
  -> 刷新 GUI 机场/跑道列表
```

## 6. 格式识别预留

当前主库解析支持：

- `AIRPORT1`
- `AIRPORT2`
- `RWYS`
- `RWYU`
- `RWYT`
- `RWYV`

当前文件读取仍以 UTF-8 为主。后续如果要导入更多来源的 `.RWY` / `.rwy` / `.stx` 文件，应增加文件编码识别，至少考虑：

- UTF-8
- GBK / CP936
- Latin-1 / ASCII

格式识别结果应进入 `RunwayFileProfile`，避免导入时丢失原始文件特征。

## 7. 非全跑道生成预留

当前第一步只复制机场完整块，不生成非全跑道数据。

后续如果要根据机场块自动生成非全跑道选择，应复用：

```text
AirportBlock.runway_ids
RunwayDataset.get_runway_ids()
StasInputBuilder._build_airport_runway()
```

当前 STAS 输入仍由用户跑道选择决定：

```text
SELECT RUNWAY ZUUU/02L,ZUUU/20R
```

但是 `APTRWY.RWY` 中会保留整个 ZUUU 机场块，确保 STAS 能找到该机场的完整跑道数据。

## 9. 2026-05-19 更新：STX 与 AIRPORT2 交叉口短跑道

当前导入入口同时支持 `.RWY`、`.rwy` 和 `.stx` 文件。三类文件内容按同一套 APTRWY 机场块格式解析，导入目标仍然是 `APTRWY_MASTER.RWY`。

当前已实现 AIRPORT2 `#INT` 交叉口短跑道生成。主库仍保留外部文件原始机场块，不直接写回生成后的短跑道；UI/校验读取主库时会在内存中生成短跑道，计算前写入 `APTRWY.RWY` 时也会再次生成同样的短跑道。

生成逻辑位于：

```text
src/stas_app/services/runway_intersection_generator.py
```

核心规则：

- 处理 AIRPORT2 的 `RWYU` 全长跑道块和其后的 `#INT` 行。
- 原始全长跑道块也会清理单引号文本行和连续 `H ` EOSID 行，保证全长跑道与交叉口短跑道显示一致。
- 根据 `#INT` 偏移量重算 TORA、TODA、ASDA，第四个距离字段保持主跑道原值。
- `RWYU` 的第一个数值使用 `#INT` 中的对正距离。
- 单引号文本行末尾的航空日期会用正则删除，例如 `19 MAY 2026`。
- 连续 `H ` EOSID 行会去掉 `H ` 前缀并用空格合并。
- 合并后的文本在基础提示词后保留 21 个空格，再拼接 EOSID 文本，输出为一条单引号文本行。

人工验证：

1. 启动桌面程序。
2. 在“机场数据导入”区域选择外部 `.RWY`、`.rwy` 或 `.stx` 文件。
3. 点击“预览”，确认新增/覆盖/跳过机场数量。
4. 选择“跳过已存在机场”或“覆盖已存在机场”。
5. 点击“导入”，确认主库写入并刷新机场列表。
6. 选择机场和跑道。
7. 点击“只算当前”或“执行队列”。
8. 确认 `runtime/stas/APTRWY.RWY` 只包含相关机场完整块。
9. 确认 STAS 计算正常。

## 10. 2026-05-19 更新：报告用特殊程序补全

STAS 原始输出 `STASOUT.out` 对 `ENG-OUT PROCEDURE` 存在固定宽度截断风险。对于 AIRPORT2 中由 `#INT` 生成或清洗后的跑道，完整 EOSID 文本已经写入运行用 `APTRWY.RWY`，因此报告导出前会额外生成一份报告读取源：

```text
STASOUT.out
  -> runway_procedure_enricher 读取 runtime/stas/APTRWY.RWY
  -> STASOUT.enriched.out
  -> Word/PDF 报告
```

规则：

- 不改写 `APTRWY_MASTER.RWY`。
- 不改写 `STASOUT.out`，保留真实 STAS 原始输出。
- 只补全 `*** SEE SPECIAL PROCEDURE FOR THIS RUNWAY ***` 对应的完整程序文本。
- 文本来源优先为运行用 `APTRWY.RWY` 中同一机场、同一跑道的完整记录。
- 连续 `H ` 行会剥离前缀并合并为一句完整 EOSID。
- 尾部航空日期会删除，例如 `19 MAY 2026`。
- `NO EMERGENCY TURN` 不做补全，沿用现有导出逻辑处理。
