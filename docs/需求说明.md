# 需求说明

## 1. 项目名称

STAS 起飞性能计算工具重构版。

## 2. 项目目标

本项目用于重构旧版 STAS 桌面工具，把现有单文件程序改造成结构清晰、可维护、可测试、可扩展的软件。

本次重构不是最终目标，而是后续持续增加功能的基础。新版架构需要支持：

- 更方便地增加新机型。
- 更方便地增加新的计算流程。
- 更方便地增加新的报告导出方式。
- 更方便地接入历史记录、批量计算、结果对比等后续功能。

旧版 `733` 机型从新版功能范围中彻底移除；旧资料保留在 `examples/` 中，仅作为历史参考，不进入新版支持列表。

## 3. 使用者

主要使用者为航务性能、运行控制、签派或相关技术人员。

使用者需要通过图形界面填写起飞性能计算参数，并获得可保存、可追溯的计算结果和报告。

## 4. 当前支持范围

新版第一阶段只支持以下机型：

- `738`
- `777F`

说明：

- 旧版中的 `777` 在新版中统一命名为 `777F`。
- 旧版中的 `733` 不再作为新版支持机型。
- 后续新增机型应通过机型配置和模板扩展，不应在 UI 或主流程中硬编码。

## 5. 输入数据

新版工具需要处理以下输入：

- 用户在界面中选择的机型。
- 用户选择或输入的机场四字码。
- 用户选择的跑道。
- 用户逐条 Scenario 选择的道面条件。
- 用户逐条 Scenario 输入的污染深度。
- 用户逐条 Scenario 选择的引气/空调状态。
- 用户逐条 Scenario 选择的防冰设置。
- 用户逐条 Scenario 选择的推力设置。
- QNH 参考值。
- 温度范围。
- 风速范围。
- 用户维护的输出顺序队列。
- 用户保存的队列方案。
- STAS 程序目录。
- STAS 机型性能数据库文件。
- `APTRWY.RWY` 跑道数据文件。
- 机型配置文件。
- STAS 输入模板文件。

后续可扩展输入：

- 批量机场/跑道列表。
- 外部 Excel/CSV 参数表。
- OFP 或航班计划数据。
- NOTAM 或跑道限制数据。
- 用户历史配置。

## 6. 输出结果

第一阶段输出：

- STAS 原始输入文件。
- STAS 原始输出文件。
- Word 报告。
- PDF 报告。
- 按 Scenario 队列顺序合并的 Word/PDF 总报告。
- 运行日志。
- 错误日志。
- Scenario 队列执行摘要。
- 队列方案存档文件。

后续可扩展输出：

- Excel 汇总表。
- 批量计算结果包。
- 结果对比报告。
- 历史记录查询结果。
- 结构化 JSON 运行元数据。

## 7. 核心功能

第一阶段核心功能：

1. 读取配置并定位 STAS 程序、模板、跑道文件和输出目录。
2. 读取机型配置，动态生成机型列表。
3. 解析 `APTRWY.RWY`，获取机场和跑道数据。
4. 校验用户输入参数。
5. 根据机型配置和模板生成 STAS 输入文件。
6. 调用 `STAS.exe` 完成起飞性能计算。
7. 保存原始输出，并生成 Word/PDF 报告。
8. 在界面中展示计算状态、耗时、输出路径和错误信息。
9. 支持 Scenario 队列驱动的输出顺序，用户可以逐条定义组合并控制先后顺序。
10. 支持不绑定机型、机场、跑道和报告格式的队列方案，方便同一套计算条件队列复用于不同任务。
11. 支持道面条件和污染深度输入，分别写入 `POPT(14)` 和 `POPT(15)`。
12. 支持队列执行完成后按 Scenario 顺序生成队列级合并 Word/PDF 报告。
13. 支持 Scenario 推力下拉选择：`738` 固定正常推力，`777F` 支持正常、10%、20% 和 `1L1BUMP`。

后续预留功能：

1. 新增机型配置。
2. 落地性能或其他性能计算模块。
3. 结果历史记录。
4. 报告模板自定义。
5. 数据版本管理。
6. 用户配置管理。

## 8. 业务规则

第一阶段业务规则：

- 新版只展示和计算 `738`、`777F`。
- 用户必须选择有效机场。
- 跑道必须来自所选机场。
- 温度范围必须符合 STAS 可识别格式。
- 风速范围必须符合 STAS 可识别格式。
- QNH 必须是有效数字。
- 机型特有参数从机型配置读取。
- STAS 输入模板不得硬编码固定绝对路径。
- 报告导出失败不应覆盖 STAS 核心计算成功状态。
- 738 和 777F 的引气、防冰等 Scenario 代码必须按各自 SCAP 文档独立映射，不得共用一套硬编码。
- 队列方案不得保存机型、机场、跑道和报告格式；温度、风、QNH 属于可保存计算条件，应随队列方案保存。
- 队列执行必须按用户界面中的 Scenario 顺序逐条执行。
- 道面条件属于 Scenario 顺序项；污染深度是道面条件的附属参数，只在积水、雪浆、干雪等需要深度的道面条件下启用。
- 推力属于 Scenario 顺序项；GUI 不再提供自由减推力输入框。
- `738` 的推力下拉框必须禁用并显示“正常”，最终写入 `{derated}=0`。
- `777F` 的推力下拉框必须提供“正常”“减推力10%”“减推力20%”“1L1BUMP”：前三项分别写入 `{derated}=0/10/20`，`1L1BUMP` 写入 `{derated}=0` 且输出 `RATING = B1L1BUMP`。

## 9. 异常情况

需要给用户清晰提示的异常包括：

- 配置文件不存在或配置项缺失。
- STAS 程序不存在。
- 模板文件不存在。
- `APTRWY.RWY` 文件不存在或解析失败。
- 用户输入为空或格式错误。
- 机场或跑道不存在。
- STAS 执行超时。
- STAS 返回错误或未生成输出文件。
- Word/PDF 报告生成失败。
- 输出目录无权限。
- SMB 跑道文件同步失败。

## 10. 界面要求

第一阶段继续使用桌面图形界面。

界面只负责：

- 用户输入。
- 参数选择。
- 按钮操作。
- 状态展示。
- 结果路径展示。
- 错误提示。
- 输出顺序队列维护。
- 队列方案保存、加载和删除。

界面层不应直接负责：

- 解析跑道文件。
- 拼接 STAS 输入文件。
- 调用外部程序。
- 生成报告。
- 保存核心配置。

这些逻辑应下沉到 `services/`、`parsers/`、`exporters/` 和 `storage/`。

说明：

- 队列方案属于轻量用户数据，当前由 `storage/scenario_order_store.py` 管理。
- 队列执行当前仍复用单次计算服务；队列级 Word/PDF 合并由 `exporters/queue_report.py` 负责，界面只接收并展示合并结果。

## 11. 技术偏好

- 默认使用 Python。
- 第一阶段桌面界面改用 DearPyGui，避免当前环境缺少 `tkinter` 导致界面无法启动。
- 不引入复杂前后端分离架构。
- 不在第一阶段引入过重技术栈。
- 核心业务逻辑必须可脱离 UI 测试。
- 新增机型优先通过配置文件和模板实现。

## 12. 验收标准

第一阶段重构完成后应满足：

- 新版界面只显示 `738` 和 `777F`。
- 旧版 `733` 不出现在新版支持列表中。
- 能解析样例 `APTRWY.RWY`。
- 能为 `738` 和 `777F` 生成 STAS 输入文件。
- STAS 输入文件中的跑道文件路径来自配置，而不是模板硬编码。
- 能生成包含 `POPT(14)`、`POPT(15)`、`CONF(4)`、`CONF(5)` 和 `POPT(25)` 的单 Scenario 输入文件。
- 参数校验能拦截明显错误输入。
- STAS 计算流程被封装在服务层。
- Word/PDF 报告生成与核心计算流程解耦。
- 用户能在 GUI 中维护 Scenario 队列并按指定顺序执行。
- 用户能保存队列方案，并在其他机型、机场、跑道和报告格式下加载复用。
- 队列执行完成后，用户能得到按队列顺序排列的合并报告。
- 至少有跑道解析、输入文件生成、参数校验的基础测试。
- 文档说明清楚后续如何增加新机型和新功能。
