# 数据流说明

本文档说明 STAS 起飞性能计算工具重构版的数据来源、处理过程、输出结果、异常流和后续扩展入口。

## 1. 总体数据流

新版主流程如下：

```text
用户操作
  -> UI 收集输入
  -> config_repository 读取应用配置
  -> app_factory 组装服务
  -> performance_service 统一编排
  -> validation 校验参数
  -> aircraft_registry 读取机型配置
  -> runway_parser 提供跑道数据
  -> input_builder 生成 STAS 输入文件
  -> stas_engine 调用 STAS.exe
  -> output_manager 保存输出和元数据
  -> exporters 生成 Word/PDF 报告
  -> UI 展示结果和错误
```

核心原则：

- UI 不直接处理复杂业务。
- 所有核心输入输出都要结构化。
- 每次计算结果应可追溯。
- 报告导出失败不应覆盖核心计算成功状态。

## 2. 机型数据流

新版第一阶段只支持：

- `738`
- `777F`

旧版 `733` 不进入新版支持范围，旧资料仅保留在 `examples/` 中。

机型数据流：

```text
config/aircraft/*.toml
  -> aircraft_registry 加载机型配置
  -> UI 显示机型列表
  -> 用户选择机型
  -> input_builder 读取对应模板和默认参数
  -> 生成 STAS 输入文件
```

后续新增机型时，优先新增：

```text
config/aircraft/新机型.toml
templates/新机型.inp
```

主流程代码不应为每个机型单独写 `if/else`。

## 3. 用户输入

第一阶段用户输入包括：

- 机型。
- 机场四字码。
- 跑道。
- 道面条件。
- 污染深度。
- 引气/空调设置。
- 防冰设置。
- 推力（顺序项）。
- QNH 参考值和 QNHREF 输出标注开关。
- 温度范围。
- 风速范围。
- 输出目录。
- 输出顺序队列。
- 队列方案存档。
- 手册起飞分析格式模板选择，属于报告输出，不属于输出顺序项；选择“不生成手册格式”时只生成临时起飞分析格式，选择具体手册模板时只生成手册起飞分析格式。

后续可扩展输入：

- 批量机场/跑道。
- Excel/CSV 参数表。
- 航班号和日期。
- OFP 数据。
- NOTAM 数据。
- 用户保存的参数方案。

## 4. 配置输入

配置来源包括：

- 全局程序配置。
- 机型配置。
- 模板目录。
- STAS 程序路径。
- STAS 工作目录。
- 跑道文件路径。
- 输出目录。
- SMB 同步配置。

配置层应由 `storage/config_repository.py` 负责读取。后续如果需要保存用户设置，再扩展写入能力。

敏感信息处理规则：

- 不在仓库中保存真实 SMB 密码。
- 示例配置只能使用占位符。
- 密码优先从环境变量或本机安全配置读取。
- 日志中不得输出密码。

当前应用级配置文件模板：

```text
config/app.example.toml
```

本地联调时建议复制为：

```text
config/app.local.toml
```

数据流：

```text
config/app.local.toml
  -> config_repository.load_app_config
  -> AppConfig
  -> app_factory.create_performance_service
  -> PerformanceService
```

## 5. 跑道数据流

跑道数据来源：

```text
APTRWY_MASTER.RWY（优先，用作完整机场主库）
APTRWY.RWY（STAS 实际读取的运行文件）
```

处理流程：

```text
APTRWY_MASTER.RWY
  -> runway_parser / runway_database_parser
  -> runway_intersection_generator 生成 AIRPORT2 #INT 交叉口短跑道
  -> AirportInfo / RunwayInfo
  -> UI 机场和跑道选择
  -> validation 校验机场和跑道
  -> 计算前从主库复制选中机场完整块
  -> runway_intersection_generator 再次生成运行文件所需交叉口短跑道
  -> 写入 runtime/stas/APTRWY.RWY
  -> input_builder 生成 SELECT RUNWAY
  -> STAS 读取 APTRWY.RWY
  -> STASOUT.out
  -> runway_procedure_enricher 按机场/跑道补全报告用特殊程序文本
  -> STASOUT.enriched.out（仅作为报告读取源，原始 STASOUT.out 保留）
```

需要解决的旧版问题：

- SMB 下载文件路径、UI 解析路径、STAS 模板路径必须统一。
- 模板中不得硬编码 `APTRWY.RWY` 的绝对路径。

新版应由配置统一提供：

```text
airport_runway_file
airport_runway_master_file
```

并在生成 STAS 输入文件时写入：

```text
AIRPORT FILE {airport_file}
SELECT RUNWAY {airport_runway}
```

规则：

- `airport_runway_master_file` 是完整机场主库。
- `airport_runway_file` 是 STAS 运行文件。
- 用户选择机场/跑道时读取主库。
- GUI 机场数据导入支持 `.RWY`、`.rwy` 和 `.stx`，文件后缀判断大小写不敏感，只写入 `airport_runway_master_file` 或同目录默认 `APTRWY_MASTER.RWY`。
- 如果主库尚不存在，导入服务会先用当前 `APTRWY.RWY` 初始化主库，再合并外部机场。
- 点击“只算当前”或“执行队列”时，系统自动把相关机场完整原始块写入 `airport_runway_file`，写入前会生成 AIRPORT2 交叉口短跑道。
- 当前只按机场复制完整块，不按用户选择的跑道裁剪机场数据。
- 主库文件尚不存在时，回退到旧逻辑，继续使用 `airport_runway_file`；导入后重新加载并切换到主库。

机场导入流程：

```text
外部 .RWY / .rwy / .stx
  -> runway_database_parser
  -> RunwayImportService.preview_import
  -> GUI 显示新增/覆盖/跳过
  -> RunwayMasterRepository 备份并写回 APTRWY_MASTER.RWY
  -> 重新加载 ApplicationContext
  -> 刷新 UI 机场/跑道列表
```

## 6. 参数校验数据流

校验入口：

```text
PerformanceRequest
  -> validation
  -> ValidatedPerformanceRequest
```

第一阶段校验规则：

- 机型必须存在于机型配置。
- 机场不能为空。
- 机场必须存在于跑道数据。
- 跑道必须属于所选机场。
- 温度范围不能为空且格式有效。
- 风速范围不能为空且格式有效。
- QNH 必须是有效数字。
- 推力设置必须属于当前机型允许选项。

校验失败时：

```text
validation error
  -> UI 显示中文错误提示
  -> 不生成 STAS 输入文件
  -> 不调用 STAS.exe
```

## 7. STAS 输入文件生成流

输入：

- 已校验请求参数。
- 机型配置。
- 机型模板。
- 跑道文件路径。

处理：

```text
PerformanceRequest
  + AircraftConfig
  + template
  -> input_builder
  -> STASINP
```

模板占位符建议：

- `{airport_file}`
- `{airport_runway}`
- `{runway_condition}`
- `{contamination_depth}`
- `{bleed_status}`
- `{anti_icing}`
- `{temperature_range}`
- `{wind_range}`
- `{qnh_ref}`
- `{qnh_ref_description}`
- `{derated}`
- `{thrust_label}`

输出：

```text
当前运行目录/STASINP
```

要求：

- 输入文件生成逻辑必须可单元测试。
- 模板替换后不应残留未处理占位符。
- 738 和 777F 应分别有测试覆盖。

## 8. STAS 执行数据流

输入：

- `STAS.exe` 路径。
- STAS 工作目录。
- `STASINP`。
- 输出文件名。

处理：

```text
stas_engine
  -> 启动 STAS.exe
  -> 传入输入文件名和输出文件名
  -> 等待执行完成
  -> 收集 stdout/stderr/STASERR
  -> 检查 STASOUT 是否生成
```

输出：

```text
StasRunResult
```

结果字段建议：

- `status`
- `raw_output_path`
- `word_report_path`
- `pdf_report_path`
- `errors`
- `warnings`
- `timings`
- `metadata_path`

当前实现位置：

- `src/stas_app/services/stas_engine.py`
- `src/stas_app/models/result.py`

异常处理：

- STAS 程序不存在：立即报错。
- STAS 超时：终止进程并保存错误。
- STAS 返回错误：读取错误信息并展示。
- STAS 未生成输出：明确提示用户。

## 8.1 单次计算编排数据流

统一入口：

```text
PerformanceService.calculate(request)
```

处理流程：

```text
PerformanceRequest
  -> validate_performance_request
  -> AircraftRegistry.get
  -> StasInputBuilder.build
  -> StasEngine.run
  -> WordReportExporter.export
  -> PDFReportExporter.export
  -> ManualTakeoffReportExporter.export（如果选择了手册模板）
  -> PDFReportExporter.export（手册 Word 转 PDF）
  -> PerformanceCalculationResult
```

规则：

- 校验失败时，不生成 STAS 输入文件，不调用 STAS。
- STAS 执行失败时，不导出 Word/PDF。
- 临时起飞分析只在 `manual_report_template_id` 为空时导出；临时 Word 导出失败时，不继续导出对应 PDF，但保留 STAS 原始输出。
- 手册起飞分析只有在 `manual_report_template_id` 不为空时导出；选择手册格式后不再同时导出临时起飞分析 Word/PDF。
- 手册模板必须匹配当前机型和推力选项，不匹配时只标记手册报告失败，不覆盖 STAS 核心计算成功状态。
- PDF 导出失败时，核心计算仍视为成功，并通过 `warnings` 返回提示。

当前实现位置：

- `src/stas_app/services/app_factory.py`
- `src/stas_app/services/performance_service.py`
- `src/stas_app/models/result.py`

## 8.2 命令行联调数据流

命令行入口：

```text
python run_stas.py calculate
```

处理流程：

```text
命令行参数
  -> PerformanceRequest
  -> config/app.local.toml
  -> AppConfig
  -> create_performance_service
  -> PerformanceService.calculate
  -> 控制台输出结果路径和警告
```

用途：

- 在桌面 UI 联调之前验证真实 `STAS.exe` 路径、工作目录和输出归档是否正确。
- 不作为长期用户界面，只作为本地联调入口。
- `run_stas.py` 会把仓库内的 `src/` 加入 Python 导入路径，避免依赖本机 `PYTHONPATH` 行为。
- `stas_engine` 会把子进程的 `TEMP` 和 `TMP` 指向 STAS 工作目录，避免 Fortran scratch 临时文件写到不可用目录。

## 8.3 DearPyGui UI 数据流

桌面入口：

```text
python run_desktop.py
```

处理流程：

```text
DearPyGui 控件
  -> PerformanceFormValues
  -> build_performance_request
  -> PerformanceRequest
  -> PerformanceService.calculate
  -> format_result_summary
  -> 界面展示输出目录、报告路径、错误和警告
```

规则：

- UI 只收集输入和展示结果。
- UI 不解析 `APTRWY.RWY`，只使用 `RunwayDataset`。
- UI 不生成 STAS 输入文件。
- UI 不直接调用 `STAS.exe`。
- UI 不直接生成 Word/PDF。
- 计算在后台线程中执行，结果通过队列回到 DearPyGui 主循环更新界面。

当前限制：

- 如果本机缺少 `dearpygui`，启动脚本会提示安装命令。
- 本轮已完成不打开窗口的控件构建检查，仍需人工点击验证完整流程。

## 8.4 Scenario 队列数据流

桌面界面当前采用 Scenario 队列驱动，不再使用全局开关笛卡尔积生成组合。

用户操作流程：

```text
用户选择机型/机场/跑道/温度/风/QNH/QNHREF 输出标注
  -> 设置当前顺序项的道面、污染深度、推力、引气、防冰
  -> 选择报告输出
  -> 点击“加入当前顺序项”
  -> 根据需要上移/下移/删除队列项
  -> 点击“执行队列”
```

执行流程：

```text
scenario_queue: list[PerformanceFormValues]
  -> build_performance_request
  -> list[PerformanceRequest]
  -> 按队列顺序逐条调用 PerformanceService.calculate
  -> list[PerformanceCalculationResult]
  -> QueueReportExporter 生成队列级合并报告
  -> UI 按队列顺序展示摘要
```

规则：

- 队列顺序由用户手动控制。
- 每条 Scenario 拥有自己的 `scenario_id`。
- `scenario_id` 会进入输出目录命名，便于追溯队列项。
- 队列执行时仍使用现有单次计算服务，保持 STAS 调用、归档、单条 Word/PDF 导出逻辑不重复实现。
- 队列执行完成后，会额外生成一个队列级合并 Word/PDF 报告。
- 道面条件是 Scenario 顺序项；污染深度随道面条件联动，只有积水、雪浆和干雪需要输入深度。
- 推力是 Scenario 顺序项；`738` 固定为“正常”并在服务层写入 `{derated}=0`，`777F` 通过机型配置映射“正常”“减推力10%”“减推力20%”“1L1BUMP”。
- 手册起飞分析格式是报告输出，不是 Scenario 顺序项；执行队列时使用界面当前选择的手册格式统一生成报告。
- 选择“不生成手册格式”时，`manual_report_template_id` 为空，服务层不会调用手册 Word/PDF 导出。
- GUI 不再传入自由 `derate`；命令行和自动化入口仍可保留 `derate` 字段用于兼容旧流程或人工联调。

当前实现位置：

- `src/stas_app/ui/desktop_app.py`
- `src/stas_app/ui/forms.py`
- `src/stas_app/services/stas_automator.py`

## 8.5 队列方案存档数据流

队列方案存档用于保存可复用的计算条件队列。它保存温度、风、QNH、QNHREF 输出标注和每条顺序项条件，不保存机型、机场、跑道和报告格式。

保存流程：

```text
scenario_queue
  -> export_order_items
  -> ScenarioOrderStore.save_order
  -> output/scenario_orders.json
```

加载流程：

```text
output/scenario_orders.json
  -> ScenarioOrderStore.load_all
  -> apply_order_items(current_form_values, saved_items)
  -> 新的 scenario_queue
```

加载规则：

- 机型使用当前界面选择。
- 机场和跑道使用当前界面选择。
- 报告输出使用当前界面选择。
- 温度范围、风速范围、QNH、QNHREF 输出标注使用队列方案里保存的值。
- 道面、污染深度、推力、引气、防冰和顺序使用队列方案里保存的值。

存档字段：

- `runway_condition`
- `contamination_depth`
- `thrust_option`
- `derate`
- `bleed`
- `anti_icing`
- `temperature_range`
- `wind_range`
- `qnh_ref`
- `describe_qnh_ref`

说明：`thrust_option` 是当前使用的推力顺序项字段；`derate` 保留用于读取旧版顺序存档，加载旧存档时会把 `10`/`20` 映射为 `777F` 的“减推力10%”/“减推力20%”。`describe_qnh_ref` 只控制 `DESCRIBE $...$` 中是否写入 `QNHREF = 当前值`；无论是否勾选，模板中的 `QNHREF = {qnh_ref}` 计算参数都会写入。`manual_report_template_id` 不再写入顺序存档。旧存档中即使存在该字段，界面也以当前“报告输出”选择为准，避免把报告格式误认为输出顺序项。

明确不保存：

- 机型。
- 机场。
- 跑道。
- 手册起飞分析格式。

当前实现位置：

- `src/stas_app/storage/scenario_order_store.py`
- `src/stas_app/ui/forms.py`

## 9. 输出保存数据流

建议每次计算创建独立输出目录：

```text
output/
  YYYY-MM-DD_HHMMSS_机场_机型/
    STASINP
    STASOUT.out
    STASOUT.enriched.out
    临时起飞分析.docx（未选择手册格式时）
    临时起飞分析.pdf（未选择手册格式时）
    手册起飞分析.docx（选择手册格式时）
    手册起飞分析.pdf（选择手册格式时）
    run_metadata.json
    status.log
    error.log
  YYYY-MM-DD_HHMMSS_QUEUE/
    STAS_QUEUE.out
    队列_临时起飞分析.docx（未选择手册格式时）
    队列_临时起飞分析.pdf（未选择手册格式时）
    队列_手册起飞分析.docx（选择手册格式时）
    队列_手册起飞分析.pdf（选择手册格式时）
```

`output_manager` 负责：

- 创建输出目录。
- 保存输入文件副本。
- 移动或复制 STAS 输出。
- 保存运行元数据。
- 避免覆盖旧结果。

当前实现位置：

- `src/stas_app/storage/output_manager.py`

`run_metadata.json` 建议记录：

- 运行时间。
- 用户输入参数。
- 机型配置文件。
- 模板文件。
- 跑道文件路径和修改时间。
- STAS 程序路径。
- 计算耗时。
- 报告导出状态。

## 10. 报告导出数据流

报告导出输入：

```text
STASOUT.out
  -> runway_procedure_enricher 读取 APTRWY.RWY 中完整机场块
  -> STASOUT.enriched.out（如果存在可补全的特殊程序）
  -> 如果未选择手册模板：WordReportExporter -> 临时起飞分析.docx -> PDFReportExporter -> 临时起飞分析.pdf
  -> 如果选择手册模板：ManualTakeoffReportExporter -> 手册起飞分析.docx -> PDFReportExporter -> 手册起飞分析.pdf
```

规则：

- 临时起飞分析 Word 失败时，保留 STAS 原始输出。
- 手册起飞分析 Word 失败时，保留 STAS 原始输出；由于手册格式与临时格式互斥，本次不会自动回退生成临时起飞分析结果。
- PDF 生成失败时，保留 Word 和 STAS 原始输出。
- `STASOUT.out` 是 STAS 原始输出，不在补全过程中改写。
- `STASOUT.enriched.out` 是报告读取源，用于补全 STAS 固定宽度截断的 AIRPORT2 `ENG-OUT PROCEDURE`。
- 补全只处理 `*** SEE SPECIAL PROCEDURE FOR THIS RUNWAY ***` 对应的完整特殊程序；`NO EMERGENCY TURN` 明确不补全，继续由现有导出逻辑跳过或保留。
- 手册模板注册表位于 `templates/reports/manual_takeoff/templates.toml`，模板文件位于同目录。
- 手册格式保留 `      ELEVATION` 前导空格。
- 手册格式每个 `ELEVATION` 段使用段首分页独立输出，并根据估算行数选择正常、紧凑或更紧凑的字号和行距，降低跨页和额外空白页风险。
- PDF 生成依赖 Windows 和 Microsoft Word COM，需要启动前检查。
- PDF 真实导出依赖 `comtypes` 和 `pywin32/pythoncom`，且 Microsoft Word COM 可能需要在用户目录写入缓存；受限沙箱内可能失败。
- 导出失败应记录到结果对象和错误日志。

当前实现位置：

- `src/stas_app/services/runway_procedure_enricher.py`
- `src/stas_app/exporters/manual_takeoff_report.py`
- `src/stas_app/exporters/word_report.py`
- `src/stas_app/exporters/pdf_report.py`

### 10.1 队列级合并报告数据流

当前队列执行完成后，界面会按 Scenario 顺序展示每条计算摘要，并额外生成一个队列级合并报告。

合并流程：

```text
list[PerformanceCalculationResult]
  -> QueueReportExporter
  -> STAS_QUEUE.out
  -> 如果未选择手册模板：队列_临时起飞分析.docx -> 队列_临时起飞分析.pdf
  -> 如果选择手册模板：队列_手册起飞分析.docx -> 队列_手册起飞分析.pdf
```

规则：

- 只合并 `succeeded == True` 且存在报告读取源的 Scenario；单次计算有 `STASOUT.enriched.out` 时优先使用补全文件，否则使用 `STASOUT.out`。
- 失败 Scenario 不阻断合并报告生成，会写入 `QueueReportResult.warnings` 并显示在界面摘要中。
- 临时起飞分析 Word 生成失败时保留 `STAS_QUEUE.out`。
- PDF 生成失败时保留 `STAS_QUEUE.out` 和已生成的 Word。
- 选择“不生成手册格式”时，队列只生成临时起飞分析格式；选择具体手册模板时，队列只生成手册起飞分析格式，不再同时生成队列临时报告。
- 手册起飞分析队列导出要求所有成功 Scenario 使用同一个手册模板；如果混合不同机型或不同推力模板，则只报错提醒，不生成手册队列报告。
- 队列报告章节顺序必须严格等于用户在 Scenario 队列中指定的顺序。

## 11. SMB 同步数据流

SMB 同步用于更新 `APTRWY.RWY`。

推荐流程：

```text
用户点击“更新跑道数据”
  -> smb_sync 读取 SMB 配置
  -> 下载临时文件
  -> 校验文件可解析
  -> 替换本地 airport_runway_file
  -> 刷新 UI 机场/跑道列表
```

不建议启动时强制下载，原因：

- 影响启动速度。
- 网络失败会阻止用户使用本地缓存。
- 状态不易解释。

启动时建议：

```text
优先读取本地 APTRWY.RWY
  -> 如果不存在，再提示用户更新或检查配置
```

## 12. 异常流

异常处理原则：

- 发现错误越早越好。
- 错误提示使用中文。
- 错误信息要说明用户下一步该怎么处理。
- 日志记录详细技术信息。
- UI 不展示过长堆栈。

常见异常流：

```text
配置错误
  -> 阻止启动核心功能
  -> 提示缺失配置项

参数错误
  -> 阻止计算
  -> 提示具体字段

STAS 执行错误
  -> 保存 STASERR/stdout/stderr
  -> 提示 STAS 返回错误

报告导出错误
  -> 标记报告失败
  -> 保留原始输出
```

## 15. 跑道过滤与非全跑道数据流

```text
APTRWY_MASTER.RWY / 外部 .RWY / .rwy / .stx
  -> runway_database_parser 读取完整机场块
  -> runway_intersection_generator 清洗 AIRPORT2 跑道文本
  -> runway_intersection_generator 拆分 #INT 和已有跑道记录中的复合交叉口
       D4-C8 -> D4-15L + C8-15L
       H4/G6-15L -> H4-15L + G6-15L
  -> 校验非全跑道标识长度 <= 8
  -> app_factory 构建 RunwayDataset
       runway_id
       record_type
       TORA
       is_intersection
  -> UI 根据当前机场和最小 TORA 过滤开关过滤可见跑道
  -> 用户只能选择当前可见跑道
  -> PerformanceService 校验机场/跑道并执行计算
```

规则：

- 最小长度过滤以 TORA 为准，只有勾选启用后才参与过滤。
- 非全跑道指 `D4-15L`、`C8-15L`、`K-15L` 这类由交叉口生成或已存在的短跑道。
- `#INT 'D4-C8'` 不再生成 `D4/C8-15L`，而是拆成 `D4-15L` 和 `C8-15L`。
- 已有 `.RWY/.stx` 中的 `H4/G6-15L` 这类复合标识会规范化为 `H4-15L` 和 `G6-15L`。
- 非全跑道标识超过 STAS 8 字符限制时直接报错，不写入主库或运行文件。
- UI 不提供全长/非全跑道类型筛选，跑道列表只显示跑道名。
- UI 过滤只影响当前可见和可选跑道，不回头修改已经加入的临时队列。
- 已保存队列方案仍不保存机场和跑道，加载时套用当前界面选择。

## 2026-05-24 单点计算数据流

单点计算使用独立页面，不进入报告/队列计算的数据流：

```text
用户切换到“单点计算”
  -> UI 收集机型、机场、单条跑道、起飞重量、OAT、风、QNH、道面、引气、防冰、推力、ATM 模式
  -> single_point_forms 生成 SinglePointTakeoffRequest
  -> SinglePointTakeoffService 校验机型、机场、跑道和数值
  -> SinglePointInputBuilder 生成 FULL 输入
  -> StasEngine 调用 STAS.exe
  -> 归档 FULL 的 STASTBL
  -> SinglePointInputBuilder 生成 ATM 输入
  -> StasEngine 调用 STAS.exe
  -> 归档 ATM 的 STASTBL
  -> stas_table_parser 解析 STASTBL
  -> 服务选择 POPT(024) 等于目标起飞重量的行
  -> 服务校验 CLIMIT(001) 与目标起飞重量是否一致，并把 CLIMIT(001) 作为结果实际 TOGW
  -> 从渲染后的输入模板 POPT 段读取 POPT(012)/POPT(013)
  -> 从当前跑道记录读取 TORA/TODA/ASDA/SLOPE
  -> UI 左侧显示 FULL/ATM 主结果，右侧显示 Runway Distance
```

字段映射：

```text
TOGW      <- CLIMIT(001)
V1        <- CLIMIT(019)
VR        <- CLIMIT(006)
V2        <- CLIMIT(005)
VREF30/VREF <- SPOUTA(005)，777F 显示 VREF30，738 显示 VREF
TO/D-TO   <- CLIMIT(028)
REDUCTION <- SPOUTA(032)
ACCEL HT  <- ACCSEG(002)
ATM TEMP  <- 最大假设温度时取 CLIMIT(044)，指定假设温度时取用户输入
AE-GO     <- CLIMIT(026) + POPT(012)
EO-GO     <- CLIMIT(023) + POPT(012)
ACCEL-STOP <- CLIMIT(024) + POPT(013)
TORA/TODA/ASDA/SLOPE <- 当前跑道 APTRWY.RWY 记录
```

数值输出规则：

- `V1/VR/V2`、`VREF30/VREF`、`ACCEL HT` 使用传统四舍五入输出整数。
- `ACCEL HT` 显示单位为 `ft AGL`。
- `TO/D-TO` 保留 1 位小数。
- `REDUCTION` 保留 1 位小数并显示百分号。
- `AE-GO`、`EO-GO`、`ACCEL-STOP` 先完成加法，再使用传统四舍五入输出整数。
- `POPT(012)`、`POPT(013)` 不从 `STASTBL` 读取，而是从本次渲染后的单点输入模板 `POPT` 段读取；当前 777F 为 `51/77`，738 为 `30/45`。
- `TORA/TODA/ASDA` 按米显示，`SLOPE` 按百分比显示。

STAS 表格输出要求：

```text
POPT(024) 对比重量,目标起飞重量
OUTPUT CLIMIT(001)
```

说明：`POPT(024)` 只表示本次变量输出请求的目标重量点；真正用于速度、距离和推力计算的重量以 `CLIMIT(001)` 为准。`FULL` 结果中如果 `CLIMIT(001)` 与输入 TOGW 不一致，服务层返回计算错误；`ATM` 结果中如果不一致，界面保留 STAS 返回的实际重量并在结果顶部显示 `The input assumed temperature could not be achieved at the requested takeoff weight`。

当前默认对比重量：

- `777F`: `25000 KG`
- `738`: `20000 KG`

### 2026-05-24 单点界面补充数据流

新增输入项进入同一条单点数据流，不进入报告/队列计算：

```text
单点页面输入
```
